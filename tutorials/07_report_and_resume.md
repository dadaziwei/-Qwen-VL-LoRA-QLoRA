# 07 项目汇报与简历描述

这一章的重点是把项目讲得像一个真实开发过程，而不是堆一串名词。面试或答辩时，别人更关心你为什么这么做、遇到什么限制、怎样取舍、后续怎么扩展。

## 1. 项目背景

我把这个项目定位成“普通开发者硬件条件下的大模型部署学习项目”。很多人手里不是 A100，而是一张 8GB 显卡。这个条件很适合学习，因为它会逼着我们认真面对模型大小、量化、Adapter 微调、推理服务这些工程问题。

项目围绕 Qwen 系列模型，完成了本地 AWQ 推理、QLoRA 轻量微调、vLLM 推理服务和 FastAPI/SSE 封装。

## 2. 技术路线

```text
Qwen 原始模型
  |
  | AutoAWQ INT4 权重量化
  v
AWQ 轻量推理模型
  |
  | PEFT + QLoRA 训练 LoRA Adapter
  v
场景适配模型
  |
  | vLLM 推理加速
  v
OpenAI 兼容推理服务
  |
  | FastAPI + SSE 封装
  v
业务系统调用
```

## 3. 可以重点讲的地方

- 为什么选 8GB 作为基准：它代表很多个人开发者的真实学习环境，能体现工程取舍能力。
- 为什么使用 AWQ：让模型更容易在普通机器上完成推理验证。
- 为什么使用 QLoRA：不全量更新模型，而是训练轻量 Adapter，符合资源友好的微调思路。
- 为什么引入 vLLM：把本地验证升级成更接近服务部署的推理引擎。
- 为什么再包一层 FastAPI：让业务接口稳定，不直接依赖底层推理框架。
- 为什么支持 SSE：生成式接口需要更好的交互体验。

## 4. 简历写法

可以写成：

```text
面向普通开发环境的大语言模型量化、QLoRA 微调与流式 API 部署
- 以 8GB 显卡作为可复现学习基准，设计本地 AWQ 推理、QLoRA 训练闭环与远程 vLLM 服务的分层方案；
- 使用 AutoAWQ 对 Qwen 系列模型进行 INT4 权重量化，降低模型部署和本地验证成本；
- 基于 PEFT + QLoRA 冻结基座模型并训练 LoRA Adapter，完成领域问答数据的轻量微调流程；
- 引入 vLLM 构建 OpenAI 兼容推理服务，将本地开发链路迁移到远程 GPU 服务环境；
- 使用 FastAPI 封装 `/v1/chat/completions` 接口，并结合 SSE 实现流式响应，方便前端或业务系统集成。
```

## 5. 答辩常见问题

问题：为什么选 8GB 显卡做项目？

回答：因为它很常见，也更有学习意义。强卡可以掩盖很多工程问题，而 8GB 环境会促使我使用量化、QLoRA、短上下文、Adapter 等更真实的部署和训练策略。

问题：为什么本地先训练小模型？

回答：学习阶段我更看重反馈速度和流程完整性。小模型能快速验证数据格式、训练脚本、LoRA 保存和接口调用。等链路稳定后，再把同样思路迁移到远程 GPU 和更大的 Qwen-VL 模型。

问题：AWQ 和 QLoRA 有什么区别？

回答：AWQ 更偏推理部署，把权重量化到 INT4；QLoRA 更偏低成本微调，用 4bit 加载基座模型，只训练 LoRA Adapter。两者分别解决部署和训练阶段的问题。

问题：为什么需要 FastAPI，直接用 vLLM 不行吗？

回答：vLLM 是推理引擎，FastAPI 是业务入口。加这一层后，可以统一接口、做鉴权日志、控制 SSE 格式，也方便以后替换底层推理框架。

问题：项目后续怎么扩展？

回答：可以补多模态 LoRA 数据处理、加入 Web 前端、做 vLLM 并发压测、封装 Docker 镜像，或者把 LoRA Adapter 合并和模型发布流程补完整。
