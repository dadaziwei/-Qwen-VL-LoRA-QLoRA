# 07 项目汇报与简历描述

本章目标：把技术实现整理成项目报告、答辩话术和简历可用表达。

## 1. 项目背景

边缘端设备显存和算力有限，直接部署 FP16 大语言模型成本高、延迟高、并发能力弱。本项目围绕 Qwen 多模态/语言模型，完成本地量化推理、低比特微调实验和 API 服务封装。

## 2. 技术路线

```text
Qwen 原始模型
  |
  | AutoAWQ INT4 权重量化
  v
AWQ 轻量推理模型
  |
  | PEFT + QLoRA 训练 LoRA Adapter
  v
场景适配模型
  |
  | vLLM 推理加速
  v
OpenAI 兼容推理服务
  |
  | FastAPI + SSE 封装
  v
业务系统调用
```

## 3. 可汇报亮点

- 面向 8GB 显存边缘端环境，制定了“本地轻量推理 + 云端训练压缩”的分层方案。
- 使用 AWQ 将模型权重量化到 INT4，降低显存占用，提升边缘端部署可行性。
- 使用 PEFT + QLoRA 冻结基座模型，只训练 LoRA Adapter，减少训练参数和优化器状态占用。
- 引入 vLLM 作为推理引擎，利用 KV cache 管理和调度能力提升并发处理表现。
- 使用 FastAPI 封装 OpenAI 风格接口，并通过 SSE 支持流式返回。
- 将 Transformers 本地后端和 vLLM 远程后端抽象成统一接口，便于部署环境切换。

## 4. 简历写法

可以写成：

```text
面向边缘端的大语言模型本地量化、轻量微调与 API 服务部署
- 面向 8GB 显存边缘端设备，设计本地 AWQ 推理、QLoRA 轻量微调与远程 vLLM 服务的分层部署方案；
- 使用 AutoAWQ 对 Qwen 系列模型进行 INT4 权重量化，降低模型加载显存与存储占用；
- 基于 PEFT + QLoRA 冻结基座模型并训练 LoRA Adapter，在低比特条件下完成领域问答数据的轻量微调；
- 引入 vLLM 构建 OpenAI 兼容推理服务，优化 KV cache 管理并提升变长文本生成的并发处理能力；
- 使用 FastAPI 封装 `/v1/chat/completions` 接口，结合 SSE 实现流式响应，支持前端或业务系统快速集成。
```

## 5. 答辩常见问题

问题：为什么本地不直接训练 Qwen-VL？

回答：8GB 显存难以稳定承载多模态模型训练。项目采用工程上更合理的分层方案，本地展示 AWQ 推理和 QLoRA 流程，完整多模态 LoRA 训练迁移到 24GB 以上远程 GPU。

问题：AWQ 和 QLoRA 的关系是什么？

回答：AWQ 主要面向推理压缩，把权重量化到 INT4；QLoRA 主要面向低显存训练，用 4bit 方式加载基座模型，只训练 LoRA Adapter。两者都服务于低资源部署，但作用阶段不同。

问题：为什么需要 FastAPI，直接用 vLLM 不行吗？

回答：vLLM 更像底层推理引擎。FastAPI 负责业务接口、鉴权、日志、参数统一、SSE 格式控制和后端切换，让应用层不依赖具体推理框架。

问题：SSE 有什么价值？

回答：生成式模型输出通常较长，SSE 可以边生成边返回，降低用户体感等待时间，也方便前端逐字显示。
